🎯 Problem Statement

The challenges MCP DevBridge was built to solve

Modern AI development workflows suffer from fragmentation: each tool (GitHub, databases, file systems) has its own proprietary API, authentication mechanism, and data format. Developers waste time writing glue code instead of building features.

  • No unified protocol for AI tools to interact with developer resources — every integration requires bespoke code.
  • Lack of real-time observability: developers cannot monitor tool calls, latency, or errors across multiple integrations in one place.
  • Security gaps when AI assistants access sensitive repositories or databases without centralized audit logging.
  • Difficult local development: spinning up individual mock servers for GitHub, databases, and file systems is time-consuming and error-prone.
  • No standardized way to expose new tools to AI models — each model vendor has a different function-calling format.

🚀 Quick Start

Get MCP DevBridge running locally in under 5 minutes

Step 1: Clone the repository

git clone https://github.com/your-org/mcp-devbridge.git cd mcp-devbridge

Step 2: Configure environment variables

cp .env.example .env # Edit .env and set your GitHub token: GITHUB_TOKEN=ghp_your_personal_access_token DB_CONNECTION=Data Source=devbridge.db

Step 3: Start all services with Docker Compose

docker compose up --build -d # Services starting: # ✓ dashboard → http://localhost:3000 # ✓ api-gateway → http://localhost:5000 # ✓ mcp-github → http://localhost:3001 # ✓ mcp-database → http://localhost:3002 # ✓ mcp-filesystem → http://localhost:3003

Step 4: Test a tool call

curl -X POST http://localhost:5000/api/mcp/call \ -H "Content-Type: application/json" \ -d '{"server":"github","tool":"list_repos","params":{}}'

Step 5: Open the dashboard at http://localhost:3000

🔒 Security Considerations

Built-in safeguards for production deployments

🔑 Token Management

GitHub tokens stored only in environment variables, never in source code or database. Tokens are never returned in API responses.

📋 Audit Logging

Every tool call is persisted with caller identity, parameters, response time, and HTTP status. Logs are immutable once written.

🗂 FileSystem Sandboxing

The FileSystem MCP server operates within a configurable root directory. Path traversal attempts (../) are blocked at the validation layer.

🛡 Query Safety

Database tool separates read queries from write queries. DDL statements (DROP, CREATE) are blocked by default in the query_table tool.

🏥 Health Monitoring

Background health checks detect server failures within 30 seconds. Unhealthy servers are excluded from routing until they recover.

🐳 Network Isolation

Docker Compose places all services on an internal network. Only the dashboard (3000) and gateway (5000) are exposed to the host.
