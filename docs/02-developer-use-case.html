Developer Guide

MCP DevBridge

Developer Use Case Guide — Step-by-step workflows, testing, and real-world usage patterns

⚡ Quick Start (Docker)

The fastest way to get the entire platform running is via Docker Compose. One command builds and starts all 5 services:

cd project-1-mcp-devbridge
docker compose up --build

Once running, access the platform at:

🌐 http://localhost:3000 — Next.js Management UI
📄 http://localhost:5000/swagger — .NET Backend Swagger UI
ⓘ First Run — Database Seeding

The first time the backend starts, it automatically seeds all 3 MCP servers (GitHub, Database, FileSystem) into the SQLite database. No manual data entry is required. Subsequent restarts will not re-seed if the records already exist.

🔧 Local Development Setup

For active development with hot-reload and debugger support, run each tier independently.

1. MCP Servers (TypeScript)

Start all 3 servers simultaneously using the npm start script:

cd mcp-servers
npm install
npm start

This launches all three on ports 3001, 3002, and 3003 in parallel. To run a single server individually for debugging:

# GitHub server only (port 3001)
cd mcp-servers
npx ts-node src/github-server.ts

# Database server only (port 3002)
npx ts-node src/database-server.ts

# FileSystem server only (port 3003)
npx ts-node src/filesystem-server.ts

2. .NET API Gateway

cd backend/McpGateway
dotnet run

The backend starts on port 5000. On startup it:

  • Creates the SQLite database (if it does not exist)
  • Seeds the 3 MCP server records
  • Begins background health checks against ports 3001–3003
  • Opens the Swagger UI at http://localhost:5000/swagger

3. Next.js Frontend

cd frontend
npm install
npm run dev

Starts on port 3000 (Next.js will use the next available port if 3000 is occupied). The frontend reads NEXT_PUBLIC_API_URL from frontend/.env.local to locate the backend.

📋 Step-by-Step Workflow

Follow these steps to go from a fresh start to running your first tool invocation through the UI:

  • 1
    Start MCP Servers

    Run npm start in the mcp-servers/ directory. Confirm all three are listening by checking http://localhost:3001/health.

  • 2
    Start .NET Backend

    Run dotnet run in backend/McpGateway. The backend seeds the DB and immediately begins health-checking all three MCP servers.

  • 3
    Start Next.js Frontend

    Run npm run dev in frontend/. Navigate to http://localhost:3000.

  • 4
    Navigate to the Servers Page

    You will see 3 seeded server entries: GitHub MCP Server, Database MCP Server, FileSystem MCP Server — each showing their URL and current status.

  • 5
    Click "Start" on Each Server

    The UI calls the backend, which pings the server's /health endpoint, sets status to Online, and syncs the tool list from /tools into the database.

  • 6
    Navigate to the Tools Page

    All 13 tools appear, grouped by their parent MCP server. Each tool shows its name, description, and required input parameters.

  • 7
    Click a Tool to Open ToolInvoker

    A modal opens with dynamically generated input fields for each parameter the tool accepts, based on its JSON schema.

  • 8
    Fill Parameters and Click "Invoke Tool"

    The frontend POSTs to the backend which proxies the call to the MCP server via JSON-RPC 2.0. The result and execution duration are displayed in the modal.

  • 9
    Navigate to the Logs Page

    Every connection event and tool invocation is logged here with its timestamp, duration, status (success / error), input parameters, and result payload.

🔹 Testing MCP Servers Directly (curl)

All MCP servers can be tested independently using curl before involving the .NET backend or the UI. This is the fastest way to validate that a server is up and a tool works.

Health & Info Endpoints

# Check if a server is alive
curl http://localhost:3001/health

# Get server metadata (name, version, description)
curl http://localhost:3001/info

# List all available tools with their schemas
curl http://localhost:3001/tools

Invoke a Tool via REST (Convenience Endpoint)

# List repositories for a GitHub user/org
curl -X POST http://localhost:3001/tools/list_repos \
  -H "Content-Type: application/json" \
  -d '{"owner": "microsoft"}'

# Describe a database table
curl -X POST http://localhost:3002/tools/describe_table \
  -H "Content-Type: application/json" \
  -d '{"table_name": "McpServers"}'

# List files in sandbox
curl -X POST http://localhost:3003/tools/list_files \
  -H "Content-Type: application/json" \
  -d '{"path": "."}'

Invoke via JSON-RPC 2.0 (Canonical Protocol)

# JSON-RPC 2.0 call to list_repos
curl -X POST http://localhost:3001/rpc \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": 1,
    "method": "tools/call",
    "params": {
      "name": "list_repos",
      "arguments": { "owner": "microsoft" }
    }
  }'
💡 Tip — Test All 3 Servers Health at Once

Run curl http://localhost:3001/health && curl http://localhost:3002/health && curl http://localhost:3003/health to confirm all three are up before starting the backend.

🎯 Use Cases

Six primary developer use cases are supported end-to-end by MCP DevBridge:

UC-01
GitHub Integration
Connect the GitHub MCP Server to list repositories, browse open issues, and view commit history for any GitHub user or organization — all without leaving the UI.
UC-02
Database Exploration
Use the Database MCP Server to list all tables in the connected SQLite database, describe their schema and column types, and run SELECT queries to inspect data.
UC-03
File System Browsing
Use the FileSystem MCP Server to list files in the sandbox directory and read file contents. Write and delete operations require ALLOW_WRITE=true.
UC-04
Tool Invocation UI
Click any tool in the Tools page to open the ToolInvoker modal. Fill in parameters in the dynamically-rendered form and see the JSON result and execution duration inline.
UC-05
Audit Trail & Logging
Every tool invocation is written to the ToolInvocations table and displayed on the Logs page with full input parameters, result payload, execution time in ms, and success/error status.
UC-06
Multi-Server Management
Start and stop individual MCP servers from the Servers page. Real-time status updates are pushed via SignalR so the UI reflects the current health without page refresh.

⚙ Required Settings

All environment variables and configuration values needed to run MCP DevBridge. These must be set before starting the respective service.

💻 Development Mode Settings

SettingWhere to SetHow to Get / NotesRequired?
GITHUB_TOKEN mcp-servers/.env or shell env GitHub.com → Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate new token. Select scopes: repo, read:user REQUIRED
ALLOW_WRITE mcp-servers/.env Set to false (default) to block write/delete. Set to true only when you need write_file or delete_file during local development. OPTIONAL
SANDBOX_DIR mcp-servers/.env FileSystem server restricts all file operations to this path. Default: ./sandbox (relative to mcp-servers directory). Create the directory if it does not exist. OPTIONAL
DB_PATH mcp-servers/.env SQLite database file path for the Database MCP Server. Default: ./data/dev.db. The file and directory are auto-created by EF Core on first run. OPTIONAL
NEXT_PUBLIC_API_URL frontend/.env.local URL of the .NET backend API. For local dev: http://localhost:5000. This is a Next.js public env var — must be prefixed with NEXT_PUBLIC_. REQUIRED
ASPNETCORE_ENVIRONMENT Shell env or backend/launchSettings.json Set to Development to enable Swagger UI, detailed error pages, and development-mode logging. OPTIONAL
ⓘ SQLite Auto-Creation

SQLite is auto-created by Entity Framework Core on the first run of the .NET backend. No dotnet ef database update or manual SQL scripts are needed in development.

🔧 Mahalliy Ishlab Chiqish Sozlamalari

cd mcp-servers

cd backend/McpGateway

Backend 5000 portida ishga tushadi. Ishga tushirishda u:

  • SQLite ma'lumotlar bazasini yaratadi (mavjud bo'lmasa)
  • 3 MCP server yozuvini joylashtiradi
  • 3001–3003 portlariga nisbatan fon sog'liq tekshiruvlarini boshlaydi

3. Next.js Frontend

cd frontend
npm install
npm run dev

3000 portida ishga tushadi (3000 band bo'lsa Next.js keyingi mavjud portni ishlatadi). Frontend backend joylashuvini topish uchun frontend/.env.local dan NEXT_PUBLIC_API_URL ni o'qiydi.

개발자 가이드

MCP DevBridge

개발자 사용 사례 가이드 — 단계별 워크플로우, 테스트, 실제 사용 패턴

⚡ 빠른 시작 (Docker)

전체 플랫폼을 실행하는 가장 빠른 방법은 Docker Compose를 통하는 것입니다. 하나의 명령으로 5개의 서비스를 모두 빌드하고 시작합니다:

cd project-1-mcp-devbridge
docker compose up --build

실행 후 다음 주소로 플랫폼에 접근합니다:

🌐 http://localhost:3000 — Next.js 관리 UI
📄 http://localhost:5000/swagger — .NET 백엔드 Swagger UI
ⓘ 최초 실행 — 데이터베이스 시딩

백엔드가 처음 시작될 때, 모든 3개의 MCP 서버(GitHub, 데이터베이스, 파일 시스템)를 SQLite 데이터베이스에 자동으로 시딩합니다. 수동 데이터 입력이 필요하지 않습니다. 레코드가 이미 존재하는 경우 이후 재시작에서는 다시 시딩하지 않습니다.

🔧 로컬 개발 설정

핫 리로드와 디버거 지원이 있는 활성 개발을 위해 각 계층을 독립적으로 실행합니다.

1. MCP 서버 (TypeScript)

npm start 스크립트를 사용하여 3개의 서버를 동시에 시작합니다:

cd mcp-servers
npm install
npm start

이렇게 하면 세 개의 서버가 포트 3001, 3002, 3003에서 병렬로 시작됩니다. 디버깅을 위해 단일 서버를 개별적으로 실행하려면:

# GitHub 서버만 (포트 3001)
cd mcp-servers
npx ts-node src/github-server.ts

# 데이터베이스 서버만 (포트 3002)
npx ts-node src/database-server.ts

# 파일 시스템 서버만 (포트 3003)
npx ts-node src/filesystem-server.ts

2. .NET API 게이트웨이

cd backend/McpGateway
dotnet run

백엔드는 포트 5000에서 시작됩니다. 시작 시 다음을 수행합니다:

  • SQLite 데이터베이스 생성 (없는 경우)
  • 3개의 MCP 서버 레코드 시딩
  • 포트 3001–3003에 대한 백그라운드 상태 확인 시작
  • http://localhost:5000/swagger에서 Swagger UI 열기

3. Next.js 프론트엔드

cd frontend
npm install
npm run dev

포트 3000에서 시작됩니다(3000이 사용 중이면 Next.js가 다음 가용 포트를 사용합니다). 프론트엔드는 백엔드를 찾기 위해 frontend/.env.local에서 NEXT_PUBLIC_API_URL을 읽습니다.

📋 단계별 워크플로우

새로운 시작부터 UI를 통해 첫 번째 도구 호출을 실행할 때까지의 단계를 따르십시오:

  • 1
    MCP 서버 시작

    mcp-servers/ 디렉토리에서 npm start를 실행합니다. http://localhost:3001/health를 확인하여 세 개가 모두 수신 대기 중인지 확인하십시오.

  • 2
    .NET 백엔드 시작

    backend/McpGateway에서 dotnet run을 실행합니다. 백엔드는 DB를 시딩하고 즉시 세 개의 MCP 서버 상태 확인을 시작합니다.

  • 3
    Next.js 프론트엔드 시작

    frontend/에서 npm run dev를 실행합니다. http://localhost:3000으로 이동합니다.

  • 4
    서버 페이지로 이동

    3개의 시딩된 서버 항목이 표시됩니다: GitHub MCP 서버, 데이터베이스 MCP 서버, 파일 시스템 MCP 서버 — 각각 URL과 현재 상태를 표시합니다.

  • 5
    각 서버에서 "Start" 클릭

    UI가 백엔드를 호출하고, 백엔드는 서버의 /health 엔드포인트를 핑하여 상태를 온라인으로 설정하고 /tools에서 도구 목록을 데이터베이스에 동기화합니다.

  • 6
    도구 페이지로 이동

    13개의 도구가 모두 상위 MCP 서버별로 그룹화되어 표시됩니다. 각 도구는 이름, 설명, 필수 입력 매개변수를 표시합니다.

  • 7
    도구 클릭으로 ToolInvoker 열기

    JSON 스키마를 기반으로 도구가 허용하는 각 매개변수에 대해 동적으로 생성된 입력 필드가 있는 모달이 열립니다.

  • 8
    매개변수 입력 후 "Invoke Tool" 클릭

    프론트엔드가 백엔드에 POST 요청을 보내고, 백엔드는 JSON-RPC 2.0을 통해 MCP 서버에 호출을 프록시합니다. 결과와 실행 시간이 모달에 표시됩니다.

  • 9
    로그 페이지로 이동

    모든 연결 이벤트와 도구 호출이 타임스탬프, 실행 시간, 상태(성공/오류), 입력 매개변수, 결과 페이로드와 함께 여기에 로깅됩니다.

🔹 MCP 서버 직접 테스트 (curl)

모든 MCP 서버는 .NET 백엔드나 UI 없이 curl을 사용하여 독립적으로 테스트할 수 있습니다. 이것이 서버가 실행 중이고 도구가 작동하는지 검증하는 가장 빠른 방법입니다.

상태 확인 및 정보 엔드포인트

# 서버가 활성 상태인지 확인
curl http://localhost:3001/health

# 서버 메타데이터 가져오기 (이름, 버전, 설명)
curl http://localhost:3001/info

# 스키마와 함께 모든 사용 가능한 도구 목록 조회
curl http://localhost:3001/tools

REST를 통한 도구 호출 (편의 엔드포인트)

# GitHub 사용자/조직의 저장소 목록 조회
curl -X POST http://localhost:3001/tools/list_repos \
  -H "Content-Type: application/json" \
  -d '{"owner": "microsoft"}'

# 데이터베이스 테이블 설명
curl -X POST http://localhost:3002/tools/describe_table \
  -H "Content-Type: application/json" \
  -d '{"table_name": "McpServers"}'

# 샌드박스 파일 목록 조회
curl -X POST http://localhost:3003/tools/list_files \
  -H "Content-Type: application/json" \
  -d '{"path": "."}'

JSON-RPC 2.0을 통한 호출 (표준 프로토콜)

# list_repos에 대한 JSON-RPC 2.0 호출
curl -X POST http://localhost:3001/rpc \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": 1,
    "method": "tools/call",
    "params": {
      "name": "list_repos",
      "arguments": { "owner": "microsoft" }
    }
  }'
💡 팁 — 3개 서버 상태 한 번에 확인

백엔드를 시작하기 전에 세 개가 모두 실행 중인지 확인하려면 curl http://localhost:3001/health && curl http://localhost:3002/health && curl http://localhost:3003/health를 실행하십시오.

🎯 사용 사례

MCP DevBridge가 종단 간 지원하는 6가지 주요 개발자 사용 사례:

UC-01
GitHub 통합
GitHub MCP 서버에 연결하여 UI를 벗어나지 않고 GitHub 사용자나 조직의 저장소 목록 조회, 열린 이슈 탐색, 커밋 히스토리 확인을 할 수 있습니다.
UC-02
데이터베이스 탐색
데이터베이스 MCP 서버를 사용하여 연결된 SQLite 데이터베이스의 모든 테이블을 나열하고, 스키마와 컬럼 유형을 설명하며, SELECT 쿼리를 실행하여 데이터를 검사할 수 있습니다.
UC-03
파일 시스템 탐색
파일 시스템 MCP 서버를 사용하여 샌드박스 디렉토리의 파일을 나열하고 파일 내용을 읽을 수 있습니다. 쓰기 및 삭제 작업에는 ALLOW_WRITE=true가 필요합니다.
UC-04
도구 호출 UI
도구 페이지의 도구를 클릭하여 ToolInvoker 모달을 엽니다. 동적으로 렌더링된 폼에 매개변수를 입력하고 JSON 결과와 실행 시간을 인라인으로 확인할 수 있습니다.
UC-05
감사 추적 및 로깅
모든 도구 호출이 ToolInvocations 테이블에 기록되며 전체 입력 매개변수, 결과 페이로드, 밀리초 단위 실행 시간, 성공/오류 상태와 함께 로그 페이지에 표시됩니다.
UC-06
다중 서버 관리
서버 페이지에서 개별 MCP 서버를 시작하고 중지합니다. 실시간 상태 업데이트가 SignalR을 통해 푸시되므로 페이지 새로 고침 없이 UI가 현재 상태를 반영합니다.

📄 백엔드 API 엔드포인트 (총 15개)

.NET 백엔드는 15개의 REST 엔드포인트를 제공합니다. 모든 도구 호출은 결과를 영속화하고 MCP 서버에 프록시하는 백엔드를 통해 흐릅니다.

메서드엔드포인트목적
GET/api/servers등록된 모든 MCP 서버 목록 조회
POST/api/servers새 MCP 서버 등록
GET/api/servers/{id}ID로 특정 서버 조회
POST/api/servers/{id}/start서버 상태 확인 + 도구 동기화
POST/api/servers/{id}/stop서버를 오프라인으로 표시
GET/api/tools모든 서버의 동기화된 모든 도구 목록 조회
GET/api/tools/{id}특정 도구의 스키마 및 메타데이터 조회
POST/api/tools/{id}/invoke매개변수로 도구 호출 — MCP 서버에 프록시
GET/api/invocations모든 도구 호출 레코드 목록 조회
GET/api/invocations/{id}특정 호출의 전체 세부 정보 조회
GET/api/logs모든 연결 및 이벤트 로그 목록 조회
GET/api/logs/server/{id}특정 서버에 범위가 지정된 로그 조회
GET/health백엔드 상태 확인 엔드포인트
GET/swaggerSwagger UI (개발 모드 전용)
WS/hubs/status실시간 서버 상태 업데이트를 위한 SignalR 허브

ⓘ SQLite 자동 생성

SQLite는 .NET 백엔드 최초 실행 시 Entity Framework Core에 의해 자동 생성됩니다. 개발 환경에서는 dotnet ef database update나 수동 SQL 스크립트가 필요하지 않습니다.

✓ 단일 서버 프로덕션에서의 SQLite
