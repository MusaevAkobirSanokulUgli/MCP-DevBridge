MCP DevBridge

Development Process & Architecture — Universal Developer Tool Integration Platform

📈 Project Overview

MCP DevBridge is a Universal Developer Tool Integration Platform built on the Model Context Protocol (MCP). It solves the real-world problem of fragmented, ad-hoc, unobservable, and insecure AI-to-tool integrations by providing a standardised, auditable gateway between AI frontends and backend developer tools (GitHub, databases, file systems).

Problem Statement

Before MCP DevBridge, developer tool integrations for AI workflows suffered from four core problems:

  • Fragmented — Every tool had its own bespoke integration, no standard protocol.
  • Ad-hoc — No structured invocation model; calls were fire-and-forget with no metadata.
  • Unobservable — No audit trail, no logging, no way to see what the AI called or when.
  • Insecure — Write operations unguarded, file system access unrestricted.

MCP DevBridge addresses all four by providing a JSON-RPC 2.0 based gateway with centralised logging, permission gating, and a polished management UI.

JSON-RPC 2.0 Protocol

All tool invocations follow the JSON-RPC 2.0 specification over HTTP POST /rpc:

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "list_repos",
    "arguments": { "owner": "microsoft" }
  }
}

🔒 Security Model

MCP DevBridge uses environment-variable-based permission gating rather than a complex auth system, keeping the security model transparent and auditable.

ControlMechanismDefault
Write operations (FileSystem)ALLOW_WRITE env varfalse — safe by default
File system path restrictionSANDBOX_DIR env var./sandbox relative to mcp-servers dir
GitHub API accessGITHUB_TOKEN env varRequired — no token, no GitHub calls
Database accessSQLite file path via DB_PATH./data/dev.db
⚠ Write Access Warning

Never set ALLOW_WRITE=true in production unless you have a specific reason. The default false prevents any write or delete operations on the file system, even if an AI model attempts to invoke them.

Quick Start

cd project-1-mcp-devbridge
docker compose up --build

The first run seeds all 3 MCP servers into the database automatically.

▶ Local Start Order

When running locally without Docker, services must be started in dependency order:

  1. Step 1 — MCP Servers (all three in parallel) 
    cd mcp-servers && npm install && npm start

    Starts all 3 servers on ports 3001, 3002, 3003.

  2. Step 2 — .NET API Gateway 
    cd backend/McpGateway && dotnet run

    Starts on port 5000. Seeds the database, begins health-checking MCP servers.

  3. Step 3 — Next.js Frontend 
    cd frontend && npm install && npm run dev

    Starts on port 3000 (or next available). Connects to backend at NEXT_PUBLIC_API_URL.

문제 정의

MCP DevBridge 이전에는 AI 워크플로우를 위한 개발자 도구 통합에 네 가지 핵심 문제가 있었습니다:

  • 단편적 — 각 도구마다 별도의 맞춤형 통합이 있었으며, 표준 프로토콜이 없었습니다.
  • 비체계적 — 구조화된 호출 모델이 없었고, 호출은 메타데이터 없이 단발성으로 이루어졌습니다.
  • 관찰 불가능 — 감사 추적, 로깅, AI가 무엇을 언제 호출했는지 확인할 방법이 없었습니다.
  • 보안 취약 — 쓰기 작업이 보호되지 않았고, 파일 시스템 접근이 제한되지 않았습니다.

MCP DevBridge는 중앙화된 로깅, 권한 게이팅, 세련된 관리 UI를 통해 이 네 가지 문제를 모두 해결합니다.

⚙ MCP 서버 (총 13개 도구)

세 가지 MCP 서버 모두 편의 REST 엔드포인트(POST /tools/:name)와 표준 JSON-RPC 2.0 엔드포인트(POST /rpc)를 통해 도구를 제공합니다. 각 서버는 GET /health와 GET /info도 제공합니다.

포트 3001
GitHub MCP 서버
list_repos get_repo list_issues create_issue get_commits
포트 3002
데이터베이스 MCP 서버
list_tables describe_table execute_query insert_record
포트 3003
파일 시스템 MCP 서버
list_files read_file write_file delete_file

JSON-RPC 2.0 프로토콜

모든 도구 호출은 HTTP POST /rpc를 통해 JSON-RPC 2.0 사양을 따릅니다:

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "list_repos",
    "arguments": { "owner": "microsoft" }
  }
}

🗃 데이터베이스 스키마

SQLite는 Entity Framework Core로 관리되는 영속성 계층으로 사용됩니다. 데이터베이스는 최초 실행 시 자동 생성되며 수동 마이그레이션 작업이 필요하지 않습니다.

테이블목적주요 컬럼
McpServers등록된 MCP 서버 인스턴스Id, Name, BaseUrl, Status, LastHealthCheck
ToolsMCP 서버에서 동기화된 도구Id, ServerId, Name, Description, InputSchema
ToolInvocations모든 도구 호출의 감사 로그Id, ToolId, Parameters, Result, Duration, Status, Timestamp
ConnectionLogs서버 상태/연결 이벤트Id, ServerId, EventType, Message, Timestamp

MCP DevBridge는 복잡한 인증 시스템 대신 환경 변수 기반 권한 게이팅을 사용하여 보안 모델을 투명하고 감사 가능하게 유지합니다.

제어메커니즘기본값
쓰기 작업 (파일 시스템)ALLOW_WRITE 환경 변수false — 기본적으로 안전
파일 시스템 경로 제한SANDBOX_DIR 환경 변수./sandbox mcp-servers 디렉토리 기준

⚠ 쓰기 접근 경고

특별한 이유가 없는 한 프로덕션에서 ALLOW_WRITE=true를 설정하지 마십시오. 기본값 false는 AI 모델이 파일 시스템의 쓰기 또는 삭제 작업을 호출하려 해도 이를 차단합니다.

🌎 Docker 아키텍처

전체 플랫폼은 5개 서비스 Docker Compose 스택으로 제공됩니다. 모든 서비스는 내부 mcp-network 브릿지 네트워크에 있습니다. 프론트엔드(3000), 백엔드(5000), 세 개의 MCP 서버(3001-3003)만 호스트에 노출됩니다.

빠른 시작

cd project-1-mcp-devbridge
docker compose up --build

첫 번째 실행 시 모든 3개의 MCP 서버가 자동으로 데이터베이스에 시딩됩니다.

▶ 로컬 시작 순서

Docker 없이 로컬에서 실행할 때는 의존성 순서에 따라 서비스를 시작해야 합니다:

  1. 1단계 — MCP 서버 (세 개 병렬 실행) 
    cd mcp-servers && npm install && npm start

    포트 3001, 3002, 3003에서 세 개 서버 모두 시작합니다.

  2. 2단계 — .NET API 게이트웨이 
    cd backend/McpGateway && dotnet run

    포트 5000에서 시작합니다. 데이터베이스를 시딩하고 MCP 서버 상태 확인을 시작합니다.

  3. 3단계 — Next.js 프론트엔드 
    cd frontend && npm install && npm run dev

    포트 3000(또는 다음 가용 포트)에서 시작합니다. NEXT_PUBLIC_API_URL을 통해 백엔드에 연결합니다.

ⓘ SQLite 자동 생성

SQLite는 .NET 백엔드 최초 실행 시 Entity Framework Core에 의해 자동 생성됩니다. 개발 환경에서는 dotnet ef database update나 수동 SQL 스크립트가 필요하지 않습니다.
